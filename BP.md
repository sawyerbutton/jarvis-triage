## 一、一个场景

早上8:15，小林戴着 Even Realities G1 眼镜出门，步行去地铁站。

昨晚他让 Claude Code 为团队的核心服务设计一套 JWT 认证迁移方案。方案在凌晨3点跑完了——一个7步的完整实施计划，包含数据库变更、中间件重构和部署策略。

以前，他需要坐到电脑前，花20分钟逐行阅读这个方案，然后逐步审批。但今天不一样。

他的耳机里传来一段简洁的语音：

> *"认证迁移方案已生成，共7步，需要你做2个决策。第一个：Token存储方式——Cookie，更安全但CORS配置复杂；还是 LocalStorage，更简单但有XSS风险。"*

小林看了一眼眼镜上的4行显示：

```
🔧 JWT迁移 (7步)
❓ 决策 1/2: Token存储
  A: Cookie (安全/CORS复杂)
  B: LocalStorage (简单/XSS风险)
```

"Cookie。"他说。

眼镜刷新：

```
✅ 决策1: Cookie
❓ 决策 2/2: 刷新策略
  A: 滑动窗口 (体验好)
  B: 固定过期 (实现简单)
```

"滑动窗口。"

```
⚠️ 风险提示: 需备份用户表
🔧 JWT迁移 → 全部批准?
  ✅ 确认执行
  ❌ 暂缓
```

"确认。"

8:17。小林刚走到地铁站，方案已经开始执行了。全程2分钟，他甚至没有掏出手机。

**这就是 Jarvis Mode。**

## 二、痛点：AI很强，但被屏幕困住了

### 2.1 AI Agent 的输出悖论

2026年，AI Agent 正在重塑软件开发。Claude Code 能自主规划复杂的代码方案，GitHub Copilot 生成了开发者 46% 的代码，AI编程助手市场已达 **73.7亿美元**。

但有一个被忽视的矛盾：**AI越强大，它的输出越冗长，越依赖屏幕交互。**

一个典型的 Claude Code 方案输出动辄数百行——架构分析、步骤分解、风险评估、备选方案。这些信息对决策至关重要，但它们被设计为在终端窗口、IDE或消息应用中阅读。

**开发者的一天中，有多少时间是坐在屏幕前的？**

通勤路上、会议间隙、午餐时间、散步思考——这些"碎片时间"加起来，占据了一天的30-40%。在这些场景中，AI生成的方案只能等待，开发者的决策被延迟，团队的执行被阻塞。

### 2.2 "信息压缩层"的缺失

问题不在AI的能力，而在**人机交互的最后一公里**。

| 场景 | 屏幕交互 | 语音+AR交互 |
|------|---------|------------|
| 审批一个代码方案 | 20分钟阅读+决策 | 2分钟语音+确认 |
| 查看CI构建结果 | 打开浏览器→找到PR→查看状态 | "构建通过，0个失败" |
| 回应Claude的技术选型问题 | 切换窗口→理解上下文→回复 | 语音听取→一句话回复 |

缺失的不是AI的智能，而是一个**信息压缩层**——将AI的复杂输出，转化为适合语音和最小化显示的格式。

### 2.3 硬件已就绪，软件在等待

AR眼镜市场正在爆发。AI智能眼镜市场预计2035年将达到 **45.9亿美元**，AR眼镜出货量在2025年上半年同比增长 **110%**。

Even Realities G1 已经证明：轻量、时尚、日常佩戴的AR眼镜是可行的。AugmentOS 提供了开源的跨设备开发SDK。

**硬件准备好了。但能在AR上运行的"AI决策交互"软件，几乎不存在。**

## 三、解决方案：Jarvis Triage — AI的信息压缩引擎

### 3.1 核心理念

Jarvis Triage 是一个**信息分诊系统**，它位于 AI Agent 和用户之间，自动将复杂的AI输出压缩为适合语音和AR HUD显示的格式。

```
┌─────────────────────────────────────────────────┐
│                  AI Agent 输出                    │
│  (Claude Code方案 / CI结果 / 代码审查 / 告警)     │
└──────────────────────┬──────────────────────────┘
                       │
              ┌────────▼────────┐
              │  Jarvis Triage  │
              │  信息分诊引擎    │
              │                 │
              │  分级 → 压缩 →  │
              │  决策提取 →     │
              │  风险标注       │
              └───┬─────────┬──┘
                  │         │
          ┌───────▼──┐  ┌──▼───────┐
          │ 🔊 语音   │  │ 📱 HUD   │
          │ ≤30秒     │  │ ≤4行     │
          │ 自然对话   │  │ 20字/行  │
          └──────────┘  └──────────┘
```

### 3.2 五级分诊系统

不是所有信息都值得打扰用户。Jarvis Triage 将AI输出自动分为五个级别：

| 级别 | 名称 | 场景 | 处理方式 |
|------|------|------|---------|
| **Level 0** | 静默 | 纯执行结果，无需用户参与 | 不输出 |
| **Level 1** | 通知 | 用户应知晓，但无需行动 | 1行HUD + 1句语音 |
| **Level 2** | 快速决策 | 2-3个选项的简单选择 | 2-3行HUD + 简短语音 |
| **Level 3** | 信息决策 | 需要上下文的复杂比较 | 3-4行HUD + 30秒语音简报 |
| **Level 4** | 方案审批 | 完整的实施方案审批 | 结构化串行审批流 |

**核心原则：用户在走路。** 一切设计都围绕这个前提。

### 3.3 杀手级功能：Level 4 串行决策流

当 Claude Code 生成一个复杂方案时，Jarvis Triage 执行四步处理：

**第一步：结构提取**
- 识别方案总步骤数
- 分析步骤间依赖关系
- 每个步骤压缩为一句话

**第二步：决策点提取**
- 找出需要用户做技术选择的步骤
- 为每个决策生成2-3个选项
- 标注每个选项的核心差异

**第三步：风险识别**
- 标记破坏性操作（文件删除、数据库变更）
- 标记前置条件（需要备份、需要停服）
- 标记不可逆操作

**第四步：串行语音+HUD审批**
- 一次只呈现一个决策
- 用户语音回答后，自动推进到下一个
- 所有决策完成后，展示风险摘要和最终确认

**为什么是"串行"而非"并行"？** 因为用户在走路，认知负荷有限。一次一个问题，是语音交互中最高效的决策模式。

### 3.4 技术架构：纯Prompt，零代码

Jarvis Triage 的整个分诊逻辑**由Prompt驱动**，无需任何传统代码。

这意味着：
- **即插即用**：一行命令安装到 OpenClaw
- **零维护成本**：随AI模型升级自动增强
- **天然跨平台**：Prompt不依赖特定运行时
- **社区友好**：任何人都可以贡献和改进分诊规则

```bash
# 安装
ln -s /path/to/jarvis-triage ~/.openclaw/skills/jarvis-triage
# 使用
"Jarvis, triage this" 或 "帮我审批一下这个plan"
```

## 七、商业模式探讨

作为一个早期开源项目，我们提出以下几种可能的商业模式路径供探讨：

### 模式A：纯开源 + 生态影响力

```
核心价值：社区声誉 → 生态话语权 → 间接变现
```

- Jarvis Triage 完全开源（MIT）
- 通过成为 OpenClaw 生态的核心组件建立影响力
- 潜在变现路径：咨询、培训、企业定制
- **优势**：社区增长快，无用户获取成本
- **风险**：直接收入不确定

### 模式B：开源核心 + SaaS增值

```
核心价值：免费核心 → 付费托管 → 企业功能
```

| 层级 | 内容 | 定价 |
|------|------|------|
| **Free** | Jarvis Triage 核心分诊（本地运行） | 免费 |
| **Pro** | 云端托管 + 语音合成 + 分诊分析 | $9.9/月 |
| **Team** | 团队协作 + 审批流转 + 权限管理 | $29.9/人/月 |
| **Enterprise** | 私有部署 + 定制分诊规则 + SLA | 定制报价 |

- **优势**：可预测收入，企业市场天花板高
- **风险**：需要基础设施投入，SaaS运营成本

### 模式C：硬件联名 + 解决方案

```
核心价值：AR眼镜最佳实践 → 硬件合作 → 联合营销
```

- 与 Even Realities、AugmentOS 等合作
- 成为"AR眼镜开发者首选AI工具"
- 联名销售、预装合作、SDK集成
- **优势**：差异化定位，高壁垒
- **风险**：依赖硬件伙伴，市场教育成本高

### 模式D：混合路径（推荐探讨）

```
Phase 0-1: 纯开源 → 建立社区和口碑
Phase 2:   SaaS增值 → 语音+AR云端服务收费
Phase 3+:  硬件合作 → AR生态深度绑定
```

**建议：从纯开源起步，在用户规模达到一定水平后，逐步引入SaaS层和硬件合作。** 具体转折点和定价策略需要根据社区反馈和市场数据动态调整。

4. **中文优先** — 面向被英语AI工具忽视的中文开发者市场

## 十、愿景：让AI从屏幕中解放出来

我们正处于一个历史性的交汇点：

- **AI Agent** 正在变得足够聪明，可以自主完成复杂任务
- **语音AI** 正在变得足够自然，可以取代打字交互
- **AR设备** 正在变得足够轻便，可以日常佩戴

**但这三股力量之间，缺少一个连接器。**

Jarvis Mode 要做的，就是成为这个连接器——把AI的智能、语音的自然、AR的便携，融合成一种新的人机交互范式。

这不只是一个开发者工具。这是对"未来我们如何与AI共事"这个问题的一种回答：

> **不是盯着屏幕等AI，而是AI跟着你走，在你需要的时刻，用最少的信息，帮你做最关键的决策。**
